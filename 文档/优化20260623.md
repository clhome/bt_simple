# 御风面板性能瓶颈分析与优化方案

## 背景

点击"网站"和"文件"菜单时有明显卡顿感。通过对前端（Pjax切换机制、JS加载策略、DOM操作）和后端（路由处理、数据查询、全局变量计算）的全面审计，定位出以下性能瓶颈。

## 🔴 P0 · 最严重的性能杀手

### 1. 网站列表 `/site/list` — 经典 N+1 查询问题

> [!CAUTION]
> 这是"网站"菜单卡顿的最核心原因。每个站点都会触发 3 次独立的文件I/O + 证书解析操作。

[site.py:56-76](file:///f:/git/gitea20250909/bt_simple/web/admin/site/site.py#L56-L76)：

```python
for site in info['list']:
    # ① 读取并正则解析 nginx vhost 配置文件 → 磁盘 I/O
    php_info = MwSites.instance().getSitePhpVersion(site_name)

    # ② 读取 vhost + 读取证书文件 + 解析X509证书 → 3~4次磁盘 I/O
    ssl_info = MwSites.instance().getSsl(site_name, '')

    # ③ os.path.exists + os.path.getsize → 磁盘 I/O
    log_path = mw.getLogsDir() + '/' + site_name + '.log'
```

**10个站点 = 至少 40+ 次磁盘I/O + 多次证书X509解析**，这是造成数百毫秒到秒级延迟的根因。

#### 优化方案

```python
# 方案A（推荐）：批量预读 + 单次循环
# 先一次性读取所有 vhost 配置内容到字典，再循环处理
vhost_dir = mw.getServerDir() + '/openresty/nginx/conf/vhost'
vhost_cache = {}
if os.path.isdir(vhost_dir):
    for f in os.listdir(vhost_dir):
        if f.endswith('.conf'):
            name = f[:-5]
            vhost_cache[name] = mw.readFile(os.path.join(vhost_dir, f))

for site in info['list']:
    site_name = site['name']
    conf_content = vhost_cache.get(site_name, '')

    # PHP 版本：直接用正则从已缓存的conf中提取
    php_match = re.search(r'enable-php-(.*)\.conf', conf_content)
    site['php_version'] = php_match.group(1) if php_match else '00'

    # SSL 天数：只检查证书文件是否存在并读取过期时间（不需要完整 getSsl）
    cert_path = ssl_dir + '/' + site_name + '/fullchain.pem'
    if os.path.exists(cert_path):
        cert_data = mw.getCertName(cert_path)
        site['ssl_days'] = cert_data.get('endtime', -1) if cert_data else -1
    else:
        site['ssl_days'] = -1

    # 日流量：保持不变（单次stat调用很快）
```

**预期提升：10个站点场景，响应时间从 ~800ms 降至 ~100ms**

### 2. `getGlobalVar()` — 每次页面请求执行 15+ 次数据库查询

> [!WARNING]
> 这个函数在 Flask 的 `@app.context_processor` 中被调用，意味着**每个页面请求**都会执行它。

[config.py:75-130](file:///f:/git/gitea20250909/bt_simple/web/utils/config.py#L75-L130)：

| 调用                                              | 代价             |
| ------------------------------------------------- | ---------------- |
| `thisdb.getOption('title')`                       | SQLite 查询      |
| `thisdb.getOption('server_ip')`                   | SQLite 查询      |
| `thisdb.getOption('site_path')`                   | SQLite 查询      |
| `thisdb.getOption('backup_path')`                 | SQLite 查询      |
| `thisdb.getOption('admin_path')`                  | SQLite 查询      |
| `thisdb.getOption('debug')`                       | SQLite 查询      |
| `thisdb.getOption('admin_close')`                 | SQLite 查询      |
| `thisdb.getSitesCount()`                          | SQLite 查询      |
| `mw.getHostPort()`                                | 读取文件         |
| `mw.M('users').where(...).getField('name')`       | SQLite 查询      |
| `thisdb.getOption('unauthorized_status')`         | SQLite 查询      |
| `thisdb.getOptionByJson('basic_auth')`            | SQLite 查询      |
| `thisdb.getOptionByJson('two_step_verification')` | SQLite 查询      |
| `mw.execShell('date +...')`                       | **子进程调用！** |
| 4×`thisdb.getOptionByJson(...)`                   | 4次 SQLite 查询  |
| `thisdb.getOption('use_cdn')`                     | SQLite 查询      |

**总计：15+ SQLite 查询 + 1次 `execShell` (fork进程)**

#### 优化方案

```python
# 方案：使用带 TTL 的内存缓存，这些配置项几乎不会变动

import time

_global_var_cache = None
_global_var_cache_time = 0
_GLOBAL_VAR_TTL = 30  # 30秒TTL

def getGlobalVar():
    global _global_var_cache, _global_var_cache_time
    now = time.time()
    if _global_var_cache is not None and (now - _global_var_cache_time) < _GLOBAL_VAR_TTL:
        return _global_var_cache.copy()

    # ... 原始逻辑 ...

    _global_var_cache = data
    _global_var_cache_time = now
    return data.copy()
```

此外，`mw.execShell('date +...')` 应该替换为 Python 原生方法：

```python
# 替换前
data['systemdate'] = mw.execShell(sformat)[0].strip()

# 替换后
from datetime import datetime
data['systemdate'] = datetime.now().strftime('%Y-%m-%d %H:%M:%S %Z %z')
```

**预期提升：Pjax 页面切换从 ~150ms 减少至 ~5ms（命中缓存时）**

### 3. 外部脚本重复加载 + 串行等待

[layout.html:453-472](file:///f:/git/gitea20250909/bt_simple/web/templates/default/layout.html#L453-L472)：

当前 Pjax 虽然使用并行加载外部脚本，但存在问题：

- `site.js`(125KB)、`files.js`(101KB) 等大文件即使已缓存在浏览器，仍然通过 `$.ajax({dataType:"script"})` 发送网络请求获取并执行
- 每个 `/static/app/*.js` 在 Pjax 切换时都被标记为 `isAppScript = true`，强制重新加载

#### 优化方案

```javascript
// 当前问题代码
var isAppScript =
  cleanSrc.indexOf("/static/app/") > -1 || cleanSrc.indexOf("/plugins/") > -1;
return isAppScript || !window.loadedScripts[cleanSrc];

// 优化：app 脚本也利用浏览器缓存，只需首次加载
// 改为使用 <script> 标签动态插入（利用浏览器 HTTP 304 缓存）
// 而非 $.ajax + eval
```

### 5. `syncPost()` — 同步 XHR 阻塞 UI

[public.js:286-299](file:///f:/git/gitea20250909/bt_simple/web/static/app/public.js#L286-L299)：

```javascript
function syncPost(path, args){
    var retData;
    $.ajax({
        type : 'post',
        url : path,
        data : args,
        async : false,  // ← 同步请求，阻塞浏览器主线程！
        ...
    });
    return retData;
}
```

在 [site.js:253](file:///f:/git/gitea20250909/bt_simple/web/static/app/site.js#L253) 中被调用：

```javascript
var www = syncPost("/site/get_root_dir");
```

同步 XHR 会完全冻结浏览器 UI（包括动画、滚动、点击），直到请求返回。

#### 优化方案

将 `syncPost` 改为异步回调或 Promise 模式，逐步消除所有同步 XHR 调用。

## 🟡 P2 · 其他优化机会

### 6. 文件管理 `getFiles()` — 每次分页都重建完整表格 HTML

[files.js:405-756](file:///f:/git/gitea20250909/bt_simple/web/static/app/files.js#L405-L756)：

`getFiles()` 函数体长达 350 行，每次调用都：

1. 发起 POST 请求获取文件列表
2. 通过字符串拼接构建所有 HTML（包括表头、工具栏、每行数据）
3. 用 `$().html()` 整体替换 DOM
4. 重新绑定所有事件（checkbox、右键菜单、拖拽选择）

对于 100 个文件的目录，这意味着约 100 个 DOM 节点的创建和 100+ 次事件绑定。

#### 优化方案

- 使用 `DocumentFragment` 或先构建完整字符串再一次性 `innerHTML`（当前已经是字符串拼接，这部分还行）
- 关键优化：**事件委托**。将事件绑定在父容器 `#fileCon` 上，而非每行每个按钮单独绑定

### 7. `setImg()` 被频繁重复调用

[index.js:487-507](file:///f:/git/gitea20250909/bt_simple/web/static/app/index.js#L487-L507)：

`setImg()` 遍历所有 `.circle` 元素并修改 CSS transform，还重新绑定 `.diskbox .mask` 的 hover 事件。它在以下位置被调用：

- `getLoad()` → 每次轮询
- `getInfo()` → 初始化
- `getNet()` → 每 3 秒轮询
- `getDiskInfo()` → 每个磁盘条目

这意味着首页每 3 秒就会多次遍历 DOM + 重新绑定事件。

#### 优化方案

```javascript
// 用防抖包装，合并 100ms 内的连续调用
var setImgTimer = null;
function setImg() {
  if (setImgTimer) clearTimeout(setImgTimer);
  setImgTimer = setTimeout(_setImgImpl, 100);
}
function _setImgImpl() {
  // ... 原始逻辑 ...
}
```

### 8. 网站列表事件绑定内存泄漏

[site.js:106-131](file:///f:/git/gitea20250909/bt_simple/web/static/app/site.js#L106-L131)：

```javascript
for (var i = 0; i < list.length; i++) {
  // ...
  $("#webBody").on("click", "#site_" + list[i].id, function () {
    // 每次 getWeb() 都重新绑定！
  });
}
```

每次翻页/搜索调用 `getWeb()` 都会在 `#webBody` 上累加新的事件监听器，造成内存泄漏和重复执行。

#### 优化方案

```javascript
// 在循环外使用事件委托，只绑定一次
$("#webBody")
  .off("click", ".setTimes")
  .on("click", ".setTimes", function () {
    var id = $(this).attr("data-ids");
    // ...
  });
```
