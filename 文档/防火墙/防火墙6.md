### 二、执行效率问题 (剩余 2 个)

1. **IP匹配仍然是线性扫描** (低危)
   - **现状** : `compare_ip()` 仍属于逐条遍历比较，IP 段数量达到十万级别时性能会明显下降。
   - **建议** : 引入 `lua-resty-ipmatcher`。

   ```
   使用 OpenResty 的 ngx.shared.DICT（共享内存）来存储一个“版本号”或“更新时间戳”。在 Lua 代码中利用 init_worker_by_lua 或者在请求阶段加一个轻量级的心跳检查：如果发现 SQLite 中的数据被面板修改了（版本号变了），就重新加载一次 IP 列表并重新构建 ipmatcher 对象。
   ```

2. **404封锁策略容易误伤** (中危)
   - **现状** : 直接判断 `404 >= 10` 就封锁 1800 秒，容易误伤前端资源发布异常或 CDN 同步失败的真实用户。
   - **建议** :
     1、如果你的 OpenResty 联动控制可以获取后缀，在计算 404 计数器时， **过滤掉图片、字体等非核心资源** （如 `.jpg`, `.png`, `.woff`, `.ico`）。

     2、改成二级报警，每分钟30次404以上，弹出页面提示，封禁1分钟（只显示警告页面）。同时如果满足每5分钟150个404则无提示直接404封禁90分钟

     ```
     内存统计：使用 ngx.shared.DICT 存储当前窗口内的计数。存储 Key 设计为 404:count:IP 和 404:uris:IP。

     URI 去重：使用一个简单的 Hash 字符串拼接在共享内存中（例如将请求的 URI 作为 Key 存入共享内存，过期时间设为 1 分钟），通过计算数量来实现去重计数。

     SQLite 联动：一级和二级完全留在 OpenResty 内存中处理，到期自动消失。
     ```

### 三、可靠性问题 (剩余 1 个)

4. **Shared Dict没有容量监控** (高危)
   - **现状** : 对 `waf_limit`、`waf_drop_ip` 等共享字典缺少 `capacity()` 和 `free_space()` 监控，遇到大规模 CC 攻击可能打满内存导致 `no memory` 使规则失效。
   - **建议** : 增加定时监控检查，一旦阈值告急即触发后台告警。

### 四、安全性问题 (剩余 2 个)

7. **缺少动态信誉系统** (高危)
   - **现状** : 缺乏对同一访客在不同阶段的风险累加判定。
   - **建议** : 建立动态信誉累加机制（如第一次异常扣 10 分，第二次扣 20 分，达到 100 分自动封禁）。

     ```
     扣分、判断全在内存（ngx.shared.DICT）中进行。只有当分数扣满 100 分，触发正式封禁时，才调用一条 SQL 语句把该 IP 写入 SQLite 的长久黑名单。
     ```

8. **缺少自动蜜罐** (中危)
   - **现状** : 无针对自动化扫描器的诱导拦截机制。
   - **建议** : 设置隐蔽的蜜罐路径（如 `/.env`、`/admin_test.php`、`/backup.zip` 等），一旦触碰该路径，信誉值直接拉满并立即封禁。支持用户自己手动添加，在全局配置中增加一个独立的“蜜罐配置”
