# 二、性能评估

## 优点

### 1. 使用 ngx.shared_dict

例如：

```
lua_shared_dict waf_limit 30m;
lua_shared_dict waf_drop_ip 10m;
lua_shared_dict waf_drop_sum 10m;
```

绝大多数统计都在共享内存完成。

避免：

* Redis
* MySQL
* SQLite

实时参与请求处理。

性能很好。

---

### 2. Worker 0 定时器设计正确

```
if ngx.worker.id() == 0 then
```

避免：

```
ngx.timer.every(...)
```

在每个 Worker 重复执行。

这是正确设计。

---

### 3. CPU监控成本低

```
ngx.timer.every(6, waf_timer_every_get_cpu)
```

每6秒更新一次。

请求阶段只读取：

```
ngx.shared.waf_limit:get("cpu_usage")
```

几乎无性能损耗。

---

### 4. GeoIP缓存

```
ngx.shared.waf_limit:set(
   "get_country"..ip,
   country,
   3600
)
```

避免每次查询 MMDB。

很好。

---

# 三、性能风险

## 风险1：大量使用 ngx.re.match

发现大量逻辑：

```
ngx.re.match(...)
```

以及：

```
C:ngx_match_list(...)
```

如果规则数量增长：

* 500条
* 1000条
* 5000条

CPU会迅速上涨。

---

建议：

预编译PCRE。

例如：

```
ngx.re.find(
    str,
    regex,
    "ijo"
)
```

使用：

```
o
```

开启 regex cache。

预计：

CPU下降 20%~40%。

---

## 风险2：POST JSON递归解析

当前：

```
extract_values(json_data)
```

递归遍历全部JSON。

如果用户上传：

```
{
  "a":{
    "b":{
      "c":{
        ...
      }
    }
  }
}
```

层级过深时：

Lua栈压力增加。

---

建议增加：

```
max_depth=10
```

限制递归深度。

---

## 风险3：文件上传读取整个Body

当前：

```
ngx.req.read_body()
local data=ngx.req.get_body_data()
```

对于：

* 50MB
* 100MB
* 500MB

上传

会造成：

* Worker内存暴涨
* GC压力增加

---

建议：

改为：

```
ngx.req.get_body_file()
```

仅检测前64KB。

不要扫描整个上传体。

---

## 风险4：GeoIP缓存Key无限增长

当前：

```
get_country{IP}
```

每个IP一个Key。

公网扫描时：

100万IP

会持续占用共享内存。

---

建议：

增加LRU机制。

或者：

```
缓存时间
3600 -> 600
```

# 五、安全风险

## 风险1：真实IP伪造

配置：

```
"trusted_proxy": [
    "127.0.0.1",
    "10.0.0.0/8"
]
```

必须严格校验：

```
X-Forwarded-For
```

来源IP是否属于可信代理。

否则：

攻击者可直接伪造：

```
X-Forwarded-For: 1.1.1.1
```

绕过封禁。

这是最高优先级检查项。

---

## 风险2：JS Challenge较弱

当前：

```
token=
waf_unbind_xxx
```

验证逻辑偏简单。

自动化工具较容易通过。

---

建议：

增加：

* 时间戳
* UA绑定
* IP绑定
* HMAC签名

例如：

```
HMAC(ip+ua+time)
```

---

## 风险3：上传检测存在绕过

当前只检查：

```
filename
mime
header
```

攻击者可使用：

```
GIF89a<?php ...
```

图片马。

---

建议增加：

文件内容规则扫描。

至少扫描：

前8KB。

---

## 风险4：IPv6支持不完整

发现代码：

```
if string.find(ip,':') then
```

大量逻辑直接跳过。

意味着：

IPv6用户可能绕过部分规则。

建议优先修复。

# 七、可靠性风险

## 风险1：共享内存耗尽

目前：

```
waf_limit 30m
```

高并发下：

* IP统计
* 国家缓存
* CC统计

都在同一个区域。

可能出现：

```
no memory
```

错误。

---

建议拆分：

```
lua_shared_dict waf_cc 50m;
lua_shared_dict waf_geo 20m;
lua_shared_dict waf_stats 20m;
lua_shared_dict waf_drop 20m;
```

避免相互影响。

---

## 风险2：日志风暴

404逻辑：

```
50次404封禁
```

很好。

但日志仍可能大量产生。

建议：

日志采样。

例如：

```
每分钟最多记录一次
```

---

## 风险3：Debug模式

发现：

```
WAF_C:setDebug(true)
```

在：

init_worker.lua

中默认开启。

生产环境必须关闭。

否则：

```
debug.log
```

会持续增长。

适用于：

* 企业官网
* CMS网站

* API服务

预计能够抵御：

* SQL注入
* XSS
