# bt_simple 面板公网部署 — 安全/可靠性/性能优化整改方案

> **审计时间**：2026-05-12
> **审计范围**：项目全部 Python 后端代码（web/、panel_task.py、panel_tools.py、setting.py）
> **前提**：该项目是服务器管理面板，登录后可访问任意目录属正常功能。
> **原则**：所有建议均在**不改变现有功能**的前提下进行加固。

## 一、安全性 (Security)

### S-01 🔴 Flask SECRET_KEY 可预测

| 项目     | 说明                                                                                                                                                             |
| -------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **位置** | `web/admin/__init__.py:69`                                                                                                                                       |
| **现状** | `SECRET_KEY` 使用 `uuid.getnode()` 生成，该值仅取决于服务器 MAC 地址，公网环境下可被探测或暴力猜解。一旦泄露，攻击者可伪造 Session Cookie 直接以管理员身份登录。 |
| **风险** | 极高 — Session 劫持 / 伪造                                                                                                                                       |
| **整改** | 首次启动时用 `os.urandom(24)` 生成随机密钥，持久化写入 `data/secret_key.pl` 文件；后续启动直接读取。                                                             |

```python
# 整改示例
secret_file = mw.getPanelDataDir() + '/secret_key.pl'
if os.path.exists(secret_file):
    app.config['SECRET_KEY'] = mw.readFile(secret_file).strip()
else:
    key = os.urandom(24).hex()
    mw.writeFile(secret_file, key)
    os.chmod(secret_file, 0o600)  # 仅 root 可读
    app.config['SECRET_KEY'] = key
```

### S-03 🔴 无 CSRF 防护

| 项目         | 说明                                                                                                             |
| ------------ | ---------------------------------------------------------------------------------------------------------------- |
| **位置**     | 全局                                                                                                             |
| **现状**     | 所有 POST 接口均无 CSRF Token 校验。攻击者可构造恶意页面诱导已登录管理员点击，触发删除文件、修改密码等高危操作。 |
| **风险**     | 高 — 跨站请求伪造                                                                                                |
| **整改方案** | 在 `requestCheck` 中对 POST 请求校验 `Referer` / `Origin` Header。简单有效，无需前端大改。                       |

```python
# web/admin/__init__.py requestCheck() 中添加
@app.before_request
def requestCheck():
    # ... 现有代码 ...

    # CSRF 防护：POST 请求校验 Referer
    if request.method == 'POST':
        # API 调用走 Header 认证，跳过
        if request.headers.get('App-Id', ''):
            return
        referer = request.headers.get('Referer', '')
        origin = request.headers.get('Origin', '')
        host = request.host
        if referer and host not in referer:
            if origin and host not in origin:
                return Response('Forbidden', status=403)
```

### S-04 🔴 缺少安全响应头

| 项目     | 说明                                                                                                                                       |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------------ |
| **位置** | `web/admin/__init__.py:139-144` `requestAfter()`                                                                                           |
| **现状** | 响应头中缺少 `X-Frame-Options`、`X-Content-Type-Options`、`Content-Security-Policy` 等安全头。面板页面可被 iframe 嵌入，易受点击劫持攻击。 |
| **风险** | 高 — Clickjacking                                                                                                                          |
| **整改** | 在 `requestAfter` 中添加安全头：                                                                                                           |

```python
@app.after_request
def requestAfter(response):
    # ... 现有代码 ...
    response.headers['X-Frame-Options'] = 'SAMEORIGIN'
    response.headers['X-Content-Type-Options'] = 'nosniff'
    response.headers['X-XSS-Protection'] = '1; mode=block'
    response.headers['Referrer-Policy'] = 'strict-origin-when-cross-origin'
    return response
```

### S-06 🟡 密码使用 MD5 存储

| 项目     | 说明                                                                                                           |
| -------- | -------------------------------------------------------------------------------------------------------------- |
| **位置** | `web/thisdb/user.py:25,93,108` / `web/admin/dashboard/login.py:149,201`                                        |
| **现状** | 用户密码仅使用 `mw.md5()` 单次哈希存储，无盐值。MD5 极易被彩虹表或暴力破解。项目已引入 `bcrypt` 依赖但未使用。 |
| **风险** | 中 — 数据库泄露后密码被秒破                                                                                    |
| **整改** | 分两步渐进迁移：① 新设密码改用 `bcrypt`；② 登录时检测旧 MD5 格式，验证通过后自动升级为 bcrypt。                |

```python
# 写入密码时
import bcrypt
def hashPassword(password):
    return bcrypt.hashpw(password.encode(), bcrypt.gensalt()).decode()

# 验证密码时 - 兼容旧 MD5
def verifyPassword(password, hashed):
    if hashed.startswith('$2b$'):  # bcrypt 格式
        return bcrypt.checkpw(password.encode(), hashed.encode())
    return mw.md5(password) == hashed  # 旧 MD5 格式兼容
```

### S-07 🟡 登录暴力破解锁定策略有副作用

| 项目     | 说明                                                                                                                                     |
| -------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| **位置** | `web/admin/dashboard/login.py:190-192,210-212,228-230`                                                                                   |

client_ip = mw.getClientIp()

if cache.get(ban_key):
    return mw.returnData(False, '该IP已被临时封禁，请1小时后重试!')

| -------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |

    return Response(status=403)
```

### S-10 🟡 Shell 命令拼接注入风险

| 项目     | 说明                                                                                                                                                    |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **位置** | `web/utils/file.py` 中的 `unzip()`、`uncompress()`、`zip()`、`closeLogs()`、`dirDelete()` 等；`web/core/mw.py` 中的 `backFile()`、`removeBackFile()` 等 |
| **现状** | 多处使用 `os.system()` 和 `mw.execShell()` 通过字符串拼接构造命令。虽然用单引号包裹参数，但若文件名包含单引号仍可注入。                                 |
| **风险** | 中 — 仅已登录管理员可触发，但公网环境下结合 CSRF 可被利用                                                                                               |
| **整改** | 对拼入命令的参数使用 `shlex.quote()` 转义：                                                                                                             |

import shlex
# 将
mw.execShell("cd '" + path + "' && tar -zxvf '" + sfile + "'")
# 改为
mw.execShell("cd " + shlex.quote(path) + " && tar -zxvf " + shlex.quote(sfile))
```

> 涉及文件较多，建议封装一个 `mw.shellQuote(s)` 统一引用 `shlex.quote`，逐步替换。

## 二、可靠性 (Reliability)

### R-01 🔴 后台线程递归式重试 — 栈溢出风险

| 项目     | 说明                                                                                                                                                                            |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **位置** | `panel_task.py:128-136,138-150,153-165,169-178`                                                                                                                                 |
| **现状** | `startPanelTask()`、`systemTask()`、`panelPluginStatusCheck()`、`check502Task()` 等函数在异常后递归调用自身重启。若异常持续发生，调用栈无限增长最终导致 `RecursionError` 崩溃。 |
| **风险** | 高 — 系统级后台进程崩溃                                                                                                                                                         |
| **整改** | 改用 `while True` 循环 + try/except，避免递归。                                                                                                                                 |

```python
# 整改前
def startPanelTask():
    try:
        while True:
            runPanelTask()
            time.sleep(5)
    except Exception as e:
        time.sleep(30)
        startPanelTask()  # 递归 ⚠️

# 整改后
def startPanelTask():
    while True:
        try:
            runPanelTask()
            time.sleep(5)
        except Exception as e:
            print('startPanelTask:', mw.getTracebackInfo())
            time.sleep(30)
```

> **需整改的函数列表**：`startPanelTask`、`systemTask`、`panelPluginStatusCheck`、`check502Task`、`openrestyAutoRestart`

## 三、性能 (Performance)

### P-01 🟡 高频文件轮询

| 项目     | 说明                                                                                                                                                |
| -------- | --------------------------------------------------------------------------------------------------------------------------------------------------- |
| **位置** | `panel_task.py:298-302,309-314`                                                                                                                     |
| **现状** | `openrestyRestartAtOnce()` 和 `restartPanelService()` 每秒轮询文件系统检查标志文件。7×24 小时运行时每天产生 ~172,800 次无效 `os.path.exists` 调用。 |
| **风险** | 低 — 无功能影响，但浪费 IO                                                                                                                          |
| **整改** | 将轮询间隔从 1 秒改为 3~5 秒，标志文件出现频率极低，不影响响应速度。                                                                                |

### P-02 🟡 os.system 可替换为原生 Python 操作

| 项目     | 说明                                                                                                                |
| -------- | ------------------------------------------------------------------------------------------------------------------- |
| **位置** | `web/utils/file.py:209-210,413-414,745-746,929` 等                                                                  |
| **现状** | `chmod`、`chown`、`rm -rf` 等操作通过 `os.system()` 调用外部命令。每次 fork 新进程开销较大。                        |
| **整改** | 非必要场景改用 `os.chmod()`、`os.chown()`、`shutil.rmtree()` 等。关键路径（如批量操作）替换后可有 ~30% 的速度提升。 |

```python
# 整改前
os.system('chmod -R 755 ' + filename)
os.system('chown -R www:www ' + filename)

# 整改后
import stat
for root, dirs, files in os.walk(filename):
    for d in dirs:
        os.chmod(os.path.join(root, d), 0o755)
    for f in files:
        os.chmod(os.path.join(root, f), 0o755)
```

## 六、从原版 mdserver-web 迁移到 bt_simple fork

### 6.1 架构兼容性分析

| 对比项              | 原版 (midoks/mdserver-web)                 | Fork 版 (bt_simple)    | 兼容性      |
| ------------------- | ------------------------------------------ | ---------------------- | ----------- |
| **安装路径**        | `/www/server/mdserver-web`                 | 相同                   | ✅ 完全兼容 |
| **数据目录**        | `data/` (SQLite + .pl 配置文件)            | 相同结构               | ✅ 完全兼容 |
| **插件目录**        | `plugins/`                                 | 相同结构               | ✅ 完全兼容 |
| **服务管理**        | `/etc/rc.d/init.d/mw` + `/usr/bin/mw`      | 相同模板               | ✅ 完全兼容 |
| **Python 虚拟环境** | `bin/`, `lib/`, `lib64/`                   | 相同                   | ✅ 直接复用 |
| **Web 服务器**      | Gunicorn + gthread                         | 相同                   | ✅ 完全兼容 |
| **数据库**          | SQLite (`data/system.db`)                  | 相同 schema            | ✅ 完全兼容 |
| **配置文件**        | `data/port.pl`, `data/ipv6.pl` 等          | 相同                   | ✅ 完全兼容 |
| **SSL 证书**        | `ssl/local/`, `ssl/nginx/`                 | 相同                   | ✅ 完全兼容 |
| **已安装软件**      | `/www/server/openresty`, `php`, `mysql` 等 | 不涉及，独立于面板代码 | ✅ 不受影响 |

> **结论**：bt_simple 是从 mdserver-web 直接 fork 的项目，目录结构、数据格式、服务管理方式完全一致。迁移本质上是**替换 Python 源代码文件**，不涉及数据库迁移、配置格式变更或服务架构调整。

**方案 A：通过 Git 仓库拉取（推荐）**

### 6.4 风险评估

| 风险项               | 等级    | 说明                                                | 应对措施                                            |
| -------------------- | ------- | --------------------------------------------------- | --------------------------------------------------- |
| **面板无法启动**     | 🟡 中   | fork 版可能引入了新依赖，或修改了 import 路径       | 检查 `panel_error.log`，缺啥依赖 `pip install` 即可 |
| **数据库不兼容**     | 🟢 低   | fork 版若新增了数据库表/字段，旧库可能缺少          | fork 本身基于相同 schema，几乎不存在此问题          |
| **插件不兼容**       | 🟢 低   | 插件独立于面板核心代码，通过 `plugins/run` 接口调用 | 插件文件未被替换，不受影响                          |
| **已有网站受影响**   | 🟢 极低 | 网站运行在 OpenResty/PHP 上，与面板代码完全独立     | 即使面板崩溃，网站依然正常运行                      |
| **原版自动更新覆盖** | 🔴 高   | `mw update` 会从原版 GitHub 拉取代码覆盖你的修改    | **必须禁用自动更新**，见下方说明                    |

### 6.5 ⚠️ 禁用原版自动更新（重要！）

迁移完成后，**必须**防止原版更新机制覆盖你的 fork 代码：

```bash
# 方法1：移除 update 命令的入口
# 编辑 /etc/rc.d/init.d/mw 或 /usr/bin/mw 脚本
# 将 mw_update() 和 mw_update_dev() 函数中的内容替换为：
mw_update(){
    echo "自动更新已禁用，请手动从 fork 仓库拉取更新"
    exit 0
}

# 方法2：让 update.sh 指向你自己的仓库（推荐）https://github.com/clhome/bt_simple
# 修改 mw.tpl 中的 GitHub URL 为你的 Gitea 仓库地址
```

### 6.6 回滚方案（万一迁移失败）

```bash
# 1. 停止面板
mw stop 2>/dev/null
# 如果 mw 命令也坏了：
ps aux | grep -E 'gunicorn|panel_task' | grep -v grep | awk '{print $2}' | xargs kill -9

cd /www/server
rm -rf mdserver-web/web mdserver-web/panel_task.py mdserver-web/panel_tools.py
tar -xzf /www/backup/mdserver-web-backup-*.tar.gz

# 3. 恢复数据库（如果被覆盖）
cp /www/backup/system.db.bak /www/server/mdserver-web/data/system.db

# 4. 重启服务
mw start
mw default
```

> 回滚耗时约 1 分钟，网站服务全程不受影响。
